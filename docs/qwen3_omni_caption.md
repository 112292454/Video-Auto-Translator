# Qwen3-Omni Captioner 评估与结论

## 概述

本文档总结了 `Qwen/Qwen3-Omni-30B-A3B-Captioner` 在 VAT 项目中的评估结果，目标是回答三个问题：

1. 它是否值得作为 VAT 的翻译辅助模块使用？
2. 它在什么场景下有帮助，什么场景下会误导？
3. 如果未来重新启用这条路线，应该如何运行和落地？

当前结论是：

- **不建议将它集成到 VAT 的主翻译链路**
- **可以把它视为一个已经验证过、但当前阶段不采用的探索方向**
- **如果以后要继续使用，应仅考虑“旁路弱摘要器”路线**

---

## 模型定位

本次评估对象：

- `Qwen/Qwen3-Omni-30B-A3B-Captioner`

它的真实能力边界是：

- `audio-only`
- 不接受文本 prompt
- 输出长篇自然语言音频描述

它更适合被理解为：

- 音频级语义摘要器
- 声音场景描述器
- 口播风格 / 情绪 / 活动类型的弱标签来源

它**不是**：

- 视觉视频 caption 模型
- 可靠的事实提取器
- 可靠的专有名词识别器
- 可直接作为翻译 prompt 强上下文的模块

---

## 研究动机

研究它的目的不是替换 VAT 现有翻译链，而是验证是否存在一种本地可运行的补充模块，为翻译提供一些额外上下文，例如：

- 当前这段是不是开场白
- 当前是不是游戏実況高压段
- 当前是不是在观察界面 / 异常 / 物体变化
- 当前是不是食评 / 开箱 / 结束语

如果这些弱语义能够稳定产出，就有机会帮助翻译阶段做：

- 风格判断
- 场景辅助判断
- 非关键的上下文补充

---

## 本地复现条件

### 机器与显卡

本次评估在多张高显存 NVIDIA GPU 的 Linux 服务器上完成。

经验上，这类模型更适合在：

- 至少一张高显存 GPU
- 最好有多张可见 GPU

的环境中运行。

### 运行环境

为了避免污染项目现有依赖，建议使用独立虚拟环境。

关键依赖版本：

- `torch 2.5.1+cu121`
- `transformers 4.57.3`
- `accelerate 1.13.0`
- `qwen-omni-utils 0.0.9`
- `decord 0.6.0`

### 模型目录

建议将模型下载到一个独立的本地目录，并在运行脚本时显式通过 `--model-path` 指定。

### 网络要求

下载阶段最大的风险不是显存，而是 Hugging Face 元信息和大文件访问的稳定性。

本次排查后确认，真正需要重点验证的不是单一下载命令，而是以下三类请求是否稳定：

1. `api/models/...`
2. `resolve/...`
3. `cas-bridge.xethub.hf.co`

也就是说，未来如果重新部署，不应直接盲试下载器，而应先确认：

- 元信息接口可访问
- 大文件重定向链路可访问
- 最终数据面可访问

## 最小运行方式

如果未来只想快速验证单个样本，可以直接运行主实验脚本。

最小前提：

- 模型目录已存在，并通过 `--model-path` 指定
- Python 环境已安装所需依赖
- 输出目录中已有待评估视频的音频产物
- 如果希望读取标题/简介等补充元信息，可选提供 `--database-path`

注意：

- 该脚本不是“下载模型并直接开始推理”的一键工具
- 它默认工作在 VAT 处理产物之上
- 至少应具备：
  - 输出目录中的 `.wav`
  - 与视频 ID 对应的目录结构

示例：

```bash
CUDA_VISIBLE_DEVICES=0,1 USE_TF=0 TRANSFORMERS_NO_TF=1 \
  python \
  scripts/qwen3_omni_caption_eval.py \
  --model-path /path/to/Qwen3-Omni-30B-A3B-Captioner \
  --output-dir /path/to/vat/output_dir \
  --database-path /path/to/vat/database.db \
  --video-id VIDEO_ID \
  --clip-duration-sec 30 \
  --work-dir /tmp/qwen3_omni_caption_eval \
  --clean-work-dir
```

如果要按真实 batch 场景批量跑，也可以通过外部清单文件驱动：

```bash
CUDA_VISIBLE_DEVICES=0,1 USE_TF=0 TRANSFORMERS_NO_TF=1 \
  python \
  scripts/qwen3_omni_caption_eval.py \
  --model-path /path/to/Qwen3-Omni-30B-A3B-Captioner \
  --output-dir /path/to/vat/output_dir \
  --database-path /path/to/vat/database.db \
  --manifest-json /path/to/manifest.json \
  --clip-duration-sec 30 \
  --window-strategy start \
  --window-strategy midpoint \
  --work-dir /tmp/qwen3_omni_caption_eval_manifest \
  --resume
```

脚本运行完成后，会在 `--work-dir` 下输出：

- `results.json`
- `results.md`

## 评估方式

### 为什么不能直接按 VAT 的 whole-batch 使用

VAT 当前默认翻译批次是：

- `100` 条字幕一个 batch

而真实 batch 的时间跨度通常是：

- `5` 到 `14` 分钟

但 Captioner 官方建议窗口长度控制在：

- `30` 秒以内

这意味着：

- **不能把一个翻译 batch 对应的整段音频直接送给 Captioner**
- 如果要用，只能先在 batch 内再选取更短窗口

### 扩展实验矩阵

本轮系统评估使用了：

- `12` 个视频
- `9` 类内容
- 每视频选择：
  - 第一批
  - 中间批
  - 最后一批
- 每个 batch 评估两种窗口策略：
  - `start`
  - `midpoint`
- 每个窗口长度：
  - `30` 秒

总样本数：

- `72`

本次扩展实验使用了一份内部样本清单。该清单主要用于复现实验矩阵，本身不是接入 VAT 所必需的组成部分。

## 运行成本

本轮 72 个样本的总体统计如下：

- 平均单样本耗时：`149.39s`
- 最快单样本耗时：`75.83s`
- 最慢单样本耗时：`255.73s`
- 总推理时间：约 `2.99` 小时

按窗口策略：

| 策略 | 样本数 | 平均耗时 |
|---|---:|---:|
| `start` | 36 | `151.52s` |
| `midpoint` | 36 | `147.26s` |

按内容类型，平均耗时差异较大：

| 内容类型 | 样本数 | 平均耗时 |
|---|---:|---:|
| 杂谈开场 | 12 | `203.06s` |
| 美食实况 | 6 | `196.92s` |
| 赛车挑战 | 12 | `151.41s` |
| 开箱评测 | 6 | `154.40s` |
| 初配信展示 | 6 | `136.51s` |
| 混合歌词叙事 | 6 | `127.09s` |
| 观察类恐怖游戏 | 6 | `123.74s` |
| 单人游戏実況 | 12 | `117.75s` |
| 恐怖游戏杂谈 | 6 | `109.57s` |

结论：

- `start` / `midpoint` 两种策略本身的成本差异不大
- 真正影响更大的变量是内容类型
- 杂谈和美食类更慢，也更容易长篇扩写

## 输出稳定性

### 重复坍缩

按“连续同字符重复超过 15 次”统计：

- 明显重复异常：`3/72`
- 比例：`4.17%`

这类结果实际不可用。

### 模板化重复

按“同视频不同窗口输出相似度 > 0.92”统计，检测到至少 2 组几乎完全重复的结果。

这说明：

- 即使换窗口
- 模型在某些视频上仍会塌缩成模板化长文

### 音乐 / BGM 偏置

粗略统计表明：

- 音乐主导式描述：`55/72`
- 比例：`76.39%`

这不是小概率问题，而是系统性偏置。

它意味着：

- 只要窗口里音乐成分比较强
- 模型就很容易先写“这是一段电子音乐 / chiptune / 游戏配乐”
- 之后再在这个前提下扩写错误上下文

## 质量判断

### 相对有帮助的场景

#### 单人游戏実況

这类内容上，Captioner 最容易抓到：

- 紧张局面
- 受伤 / 成功 / 失败反应
- 实时実況感

虽然专有名词仍然不稳，但“这是一段高强度游戏实况”这一层常常是对的。

#### 观察类恐怖游戏

这类内容上，它有时能抓到：

- 在观察变化
- 在确认异常
- 在操作界面

但对具体对象、地点、玩法机制仍然不稳。

#### 恐怖叙事 / 案件说明

这类内容上，它对“像在读案件说明 / 恐怖事件说明 / 紧张叙事”的把握相对较好，属于当前较有价值的类型。

### 帮助有限的场景

#### 开箱评测

它经常能看出“在开箱/展示物品”，但角色名、作品名、周边出处常常会乱猜。

#### 美食实况

它有时能抓到“在讨论食材 / 味道 / 吃没吃到 / 卖没卖完”这类粗主题，但仍容易被片头音乐带偏，也容易补写不存在的细节。

### 不适合作为强上下文的场景

#### 杂谈开场

能抓到：

- 自我介绍
- 开场白
- 道歉 / 结束语 / 直播互动感

但经常乱造：

- 人名
- 人设
- 背景设定
- 具体事件

#### 赛车挑战

这类内容最容易把高强度竞速音频错认成别的游戏类型，甚至出现长串重复或把内容脑补成格斗游戏 / 其他 IP。

#### 初配信展示 / 混合歌词叙事

这类内容窗口间风格漂移明显，模型常常在：

- 自我介绍
- 歌词理解
- 商品介绍
- 个人经历

之间跳来跳去，稳定性不足。

## 可用与不可用

### 当前阶段可用的部分

如果只把它当作 **弱摘要器**，当前相对可用的输出是：

- `audio_type`
  - 更像游戏実況 / 杂谈 / 观察 / 开箱 / 食评 / 结束语
- `speaker_style`
  - 情绪高、口播、実況、收尾语气
- `mood`
  - 紧张、兴奋、道歉、轻松
- `bgm_heavy`
  - 是否明显受音乐主导

这些信息最多适合作为：

- 旁路 metadata
- Web UI 展示
- debug 辅助
- 后续人工分析

### 当前阶段不可用的部分

当前明确不应该依赖它做：

- 游戏名 / IP 名识别
- 人物名 / 联动对象识别
- 视频事实摘要
- 翻译术语统一
- 直接注入翻译 prompt 的主上下文

---

## 为什么当前不采用

当前不采用，核心原因有四个。

### 1. 它不是视觉模型

很多我们真正想补的上下文，其实带有明显视觉属性：

- 画面里是谁
- 是不是多人
- 在和谁互动
- 当前画面是什么主题

Captioner 只看音频，天然做不到这部分。

### 2. 它在事实层不够稳

它在风格层经常有帮助，在事实层经常误导。

而翻译系统更需要的是：

- 稳定术语
- 稳定人物识别
- 稳定主题识别

这一点它没有达到可接入门槛。

### 3. whole-batch 不成立

VAT 真实翻译批次和 Captioner 最佳窗口不匹配，这意味着：

- 即使继续尝试
- 也必须额外实现一整套：
  - 窗口选择
  - 输出压缩
  - 异常过滤

这已经不是简单接一个模型，而是新增一个完整子系统。

### 4. 成本和收益不匹配

为了获取当前这些并不稳定的弱信息，需要付出的成本包括：

- 模型准备
- 网络下载和环境维护
- 额外推理时间
- 输出清洗与异常过滤

在当前质量下，这个收益不够高。

## 如果未来还想用，建议怎么做

唯一值得继续尝试的方向是：

### 旁路弱摘要器

不是主翻译上下文，不直接进 prompt。

建议路线：

1. **先做 batch 内窗口选择**
   - 优先选口播密度高、非纯 BGM 的短窗口

2. **再做结构化压缩**
   - 不保留原始长英文说明
   - 只提取弱标签，例如：
     - `audio_type`
     - `speaker_style`
     - `mood`
     - `topic_guess`
     - `is_intro`
     - `is_outro`
     - `is_gameplay`
     - `bgm_heavy`

3. **必须做异常检测**
   - 长串重复
   - 模板化重复
   - 音乐主导式输出
   - 超长输出

4. **只写旁路 metadata**
   - 例如：
     - `metadata['caption_context']`

5. **先只用于展示和 debug**
   - Web UI
   - 调试面板
   - 人工审阅

只有在下一轮验证中证明：

- 稳定
- 低误导
- 真能帮助翻译

才值得重新考虑接入主链路。

## 最终结论

当前阶段的正式结论是：

**Qwen3-Omni Captioner 是一个已经验证过、具有一定研究价值和少量辅助价值，但暂不采用的探索方向。**

更具体地说：

- 它有帮助，但帮助主要在弱语义层
- 它不能提供足够可靠的强上下文
- 它不能直接用于 VAT 当前的真实翻译 batch
- 它当前不值得进入主翻译链
- 如果以后要继续，应只走“旁路弱摘要器”路线
